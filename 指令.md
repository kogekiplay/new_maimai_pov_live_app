### 阶段一：打造独立的“显存回读”工具类 (最安全的第一步)

**目标**：把 `MetalTexture` 转回 `CVPixelBuffer` 的能力封装成独立类，完全不碰现有业务逻辑。

**【提供给 AI 的 Prompt】**：

> "请帮我在 `MaimaiPOV/Rendering/` 目录下新建一个纯工具类文件 `TextureReadback.swift`。
>
> 1. 它需要使用 Metal 的 `MTLBlitCommandEncoder` 将传入的 `MTLTexture` 拷贝到 CPU 可访问的共享内存中（参考目前 `MetalStabilizer.swift` 里的 `readOutputTexture` 逻辑）。
> 2. 提供一个方法 `func read(from texture: MTLTexture) -> CVPixelBuffer?`。
> 3. 注意：内部必须使用 `MTLCommandBuffer` 的 `waitUntilCompleted()` 来确保拷贝完成。
> 4. 注意：为了防止内存暴涨，请内部使用 `CVPixelBufferPool` 来循环利用 Buffer，而不是每次都 `CVPixelBufferCreate`。
>
>    **要求**：仅仅创建这个类，绝对不要修改现有项目中的任何其他文件。"

**【人工验证标准】**：

1. 编译项目，能够成功 Build。
2. 运行 App，一切表现和重构前一模一样（因为这个类还没被使用）。

> 2. 在内部定义一个后台串行队列：`let pipelineQueue = DispatchQueue(label: "com.maimai.pipeline", qos: .userInteractive)`。

### 阶段四：实装“双队列模型”（最核心的性能防线）

**目标**：解决潜在的 UI 卡顿风险，将 GPU 计算和 YOLO 移入后台专属队列，并彻底解耦 MetalView 的渲染。

**【提供给 AI 的 Prompt】**：

> "现在我们要优化 `LivePipelineManager` 中的 `camera.onVideoFrame` 闭包，解决主线程潜在的阻塞问题，引入『双队列模型』。
>
> 1. 在 `onVideoFrame` 闭包中，将 `MotionManager` 查询、`MetalStabilizer.process`、`yoloDetector.enqueue` 以及 `CropRenderer.process` 这所有的计算和命令提交过程，**全部放入** **`self.pipelineQueue.async { ... }`** **中执行**。
> 2. 当 `CropRenderer` 处理完毕后，在 `pipelineQueue` 中获取 `cropRenderer.outputTexture`。
> 3. **只把最终的纹理引用和 UI 需要更新的数据切回主线程**：
>
> Swift
>
> ```
> DispatchQueue.main.async {
>     self.previewTexture = cropRenderer.outputTexture
>     self.stabLagMs = elapsed * 1000.0
> }
>
> ```
>
> 1. 确保原来代码中所有会导致 UI 线程和相机回调队列竞争的死锁或阻塞点都被清除。"

**【人工验证标准】**：

1. 运行 App。
2. **严苛测试**：大幅度晃动手机（触发重度防抖），同时在屏幕上疯狂拖动 UI 滑块（如 FOV 或 ISO）。
3. 如果 UI 滑块非常顺滑且没有卡顿，同时视频画面依然保持高帧率（绿色的 60FPS），说明双队列改造完美成功！

**目标**：在安全的后台队列中调用阶段一做好的回读工具，准备好纯净的 `CVPixelBuffer`。

>
> 1. 在 `LivePipelineManager` 中实例化阶段一写的 `TextureReadback`。
> 2. 在 `LivePipelineManager` 中定义一个回调闭包：`var onStreamBufferAvailable: ((CVPixelBuffer, CMTime) -> Void)?`
